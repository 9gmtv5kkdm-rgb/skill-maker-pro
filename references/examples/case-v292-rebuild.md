# v2.9.2 → v3.0.0 翻车-修复全记录

1. **翻车**（2026-07-06 上午）：v2.9.2 用 203KB 单文件、5 分钟示例藏在 L1274、触发词 3 个产品化表述 → 自我解剖发现自己触犯 AP27-AP30
2. **重建**（2026-07-06 下午）：5 项核心改动（外壳从 203KB→18KB、PF 质量门、触发词 27 个用户原声型、AP27-AP30 新增、references/ 双层结构）

## 关键踩坑与经验

### 坑 1：203KB 单文件（触犯 AP27）

**证据**：SKILL.md 203KB、3819 行；5 分钟示例在 L1274。

**修复**：
```
# v2.9.2
SKILL.md (203KB)
  ├── § 1-9 全部内联
  └── 3819 行

# v3.0.0
SKILL.md (18KB)          ← 外壳：5 分钟看懂
references/
  ├── methodology/        ← 6 个方法论文档
  ├── examples/           ← 3 个完整案例
  ├── anti-patterns/      ← 30 条反模式
  ├── quality-gate/       ← PF 清单 + 专家团模板 + auto-test.py
  ├── cases/              ← 18 个常见错误
  ├── templates/          ← 5 个模板
  ├── state/              ← evolution-state.json
  └── learnings/          ← 共享 learnings
```

**经验**：外壳 14KB + references/ 索引 —— 路过的人 5 分钟看完，想深入的人按需加载。

### 坑 2：PF 8/9 自报假数据

**证据**：SKILL.md 写"8 项 PF"但表列出 9 行；evolution-state.json 写 `pf_count: 9` 但 auto-test.py 硬编码 8。

**根因**：改了 SKILL.md 文本忘改 auto-test.py；改 auto-test.py 忘改 evolution-state.json；三个文件各自维护自己的数字，无交叉校验。

**修复**：
- `evolution-state.json` 的 `pf_count` → 单一数据源
- `auto-test.py` 拉 `pf_count` 动态计数，不硬编码
- `release.py` 加 cross-file consistency check（SKILL.md vs evolution-state.json vs auto-test.py vs pf-checklist.md）

### 坑 3：版本越位（138 处 v3.0/v3.1.0/v3.2.0）

**证据**：v3.0 目标未达，却已在 SKILL.md、25 个 references/、release.py 等 138 处出现 v3.1.0/v3.2.0 计划（历史：v2.9.3 越位事件）。

**根因**：迭代太快，"计划"和"已完成"界限模糊；没有"版本目标 vs 版本完成"的 gate。

**修复**（v3.0.0）：
- 版本回退至 v3.0.0（patch 级，诚实标记）
- 所有 v3.0 标记改为"v3.0 目标未达"
- `evolution-state.json` 删除 `next_planned` 字段

### 坑 4：PowerShell 编码（3 次踩坑）

**证据**：
- 发布 v2.9.2 时 `startsWith("---")` 因 UTF-8 BOM 失败 → 用 Python zipfile 无 BOM
- release.py 报错中文字段在 PowerShell 终端乱码 → 改用纯 ASCII
- v3.0.0-upgrade-notes.md 又出现 BOM → 用 strip-bom.py 清理

**经验**：**PowerShell 永远默认带 BOM，Python 打包用 zipfile 不用 shell**。

### 坑 5：MCL dreaming 空壳

**证据**：v2.9.2/v3.0 声称有"MCL 三阶段记忆系统"，但 mcl-system.md 是 PLACEHOLDER；SKILL.md 引用 4 处但无实际实现。

**修复**（v3.0.0）：mcl-system.md 从占位升级为完整设计的"纯人工触发"版本（三阶段 Light→REM→Deep，用户主动触发，不自动修改文件）。
