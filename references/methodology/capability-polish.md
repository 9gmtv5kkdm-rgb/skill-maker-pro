# 能力层打磨方法（Capability-Layer Polish）

> 与「门面打磨」互补：门面解决"被发现、被理解"，能力层解决"真能干、干得好"。门面救不了弱能力——一个描述很专业但内部方法论空洞的技能，用户用一次就弃。

## 一、何时做能力层打磨

- 门面已达标（`displayName` / `description` / 触发词 / 首屏钩子齐备），但内部"说要做 X 却没给怎么做"。
- 用户反馈是"用着不对 / 没用 / 跑不出结果"，而不是"找不到 / 看不懂"。
- 技能含统计 / 检索 / 生成类动作，却只有声明式描述、缺真实算法或脚本。

## 二、判定能力层是否完善（精品标准）

逐条对照，任一不满足即判"部分不完善"：

1. **决策流具体**：说"做 X"给出了步骤 / 分支 / 判断标准，不是口号。
2. **有真实产物**：模板 / 检查表 / 决策表 / 参照表齐备，无 `...` 占位。
3. **边界覆盖**：空数据、极端值、歧义输入有明确处理路径。
4. **可执行实现**：计算 / 检索 / 生成类技能，关键算法有真实脚本或明确公式，非纯文字。
5. **不冗余不灌水**：每段都有用，无凑数填充。

## 三、打磨动作（只动能力，不碰门面）

- 加 `references/*.md`：检查表、决策表、术语表、场景查询模板。
- 加 `scripts/*.py`：真实"读数据 → 计算 → 出图 / 出报告"，纯标准库、带样例自检。
- 在 `SKILL.md` 加"能力索引"小节，挂接新增文件。
- **禁止改**：`displayName` / `description`（frontmatter）/ `triggers`（YAML 与正文）/ 首屏钩子 / `allowedTools` / `version`。

## 四、铁律

- **加价值不灌水**：已强的技能直接判"强、无需改"，不硬加东西凑数。
- **脚本须跑通**：新增 `scripts/*.py` 必须用样例数据 `python xxx.py` 跑通自检（含帕累托 / 交叉表 / 归一化等核心逻辑断言）。
- **改动可验证**：每段能力补充都要落到具体文件，禁止只在正文写"已增强"。
- **门面不动**：能力层打磨后，门面字段保持上一轮打磨结果，不复改。

## 五、复验清单

- 门面复验：`displayName` == 清单 `cn`、`description` 含价值动词、触发词未被删减——确认能力改动没误伤门面。
- 结构复验：抽查 2–3 个高风险改动（尤其新增脚本 / SVG 段落），确认 YAML 未破、首屏钩子仍在。
- 能力复验：跑一遍新增脚本的自检，确认输出非占位。
