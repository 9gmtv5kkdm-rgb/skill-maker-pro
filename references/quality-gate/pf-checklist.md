## 使用流程

```bash
# 1. 写完 SKILL.md
# 2. 运行自检
python auto-test.py ~/.qclaw/skills/my-skill/

# 3. 15 项内容门全 PASS（条件门按 tier 转 ADV）→ 才能发布
# 4. 任何 FAIL → 修复后重跑
```

## 缺陷等级定义（P0/P1/P2，v3.0.7 新增）

> PF 质量门判定缺陷按严重度分级，评审与发布以此为准。

| 等级 | 含义 | 发布策略 |
|---|---|---|
| **P0** | 阻断级：违反不可变核心 / 红线 / 门面硬门缺失 / 质量门脚本与规范脱节（如 auto-test 漏测某 PF） | **必须修，否则禁止发布** |
| **P1** | 重要级：影响可用性、一致性、可维护性（版本叙事过时、PF 计数口径不一、P 分级悬空） | 发布前应修，可酌情排期 |
| **P2** | 建议级：锦上添花、文档润色、非阻塞优化 | 可选，不阻塞发布 |

> 反例（v3.0.7 前真实存在，已修）：PF5 与红线#22 冲突、PF10/PF11 无自动覆盖属 **P0**；PF 计数口径不一、版本叙事过时、P 分级悬空属 **P1**。

## 门面质检规范（PF11，v3.0.4 新增）

> 从 43 个技能（31 个自制 + 12 个第三方）全量扫描中暴露的系统性通病提炼：功能都硬，门面整齐地差——无 displayName、触发词写成系统路由指令、首屏写自我描述、description 罗列功能、锁死在链路里不可独立触发。**门面是「路人怎么发现并想用」的开关，不是装饰。**

| # | 检查 | 阈值 | 说明 |
|---|---|---|---|
| **PF11a** | displayName 必填 | 存在且非 slug | 给人看的爆款名（如「合同避坑扫描」），不是内部 slug |
| **PF11b** | 触发词用户原声 | 正文「触发词」小节 ≥5 条口语 | 写「帮我看看这份合同有没有坑」，禁止 `contract-intake 锁定 review_path=` 这类系统路由指令 |
| **PF11c** | 首屏价值钩子 | frontmatter 后前 200 字出现「用户能得到什么」 | 禁止以「本技能/模块/功能」开头；第一句回答「用它能解决什么」 |
| **PF11d** | description 写收益（硬门槛） | 必须含≥1 个价值动词（帮你/解决/避免/省/一键/揪出…），否则 PR 不通过 | 写「帮你 3 分钟排出参考文献格式」，不写「本技能用于参考文献格式化」；扫描器 `desc_punch` 以 VALUE_WORDS 复测 |
| **PF11e** | 默认可独立触发 | 无前置锁死 | 链路调用是可选增强，非「必须由 X 调用才能用」 |

**与 PF4 的关系**：PF4 查 frontmatter `triggers:` 字段；PF11b 查**正文「触发词」小节**的用户原声内容——两者互补，auto-test 后续需同时抓两处（避免「触发词写在正文、PF 漏检」的通病复现）。

**去模板化（PF11f，v3.0.4 R1 反哺）**：R1 批量改造用「类型模板」把 42 技能门面拉到及格线，却暴露同质化——14 个学术系钩子句完全相同、13 个合同系完全相同。爆款门面要求**每个技能有独特可感知收益**，同类复制通用句是另一种失败。
- ❌ 坏：「论文里的繁琐活一键规范搞定」（14 个学术系通用）
- ✅ 好：「参考文献格式调不对被退稿？3 分钟按 GB/T 7714 排好，附期刊模板一键切换」
- 钩子/触发词/description 必须含**具体场景 / 数字 / 对比**之一，禁止类别通用句。

**L3 唤醒提醒（v3.0.4 R1 反哺，v3.0.6 R4 再强化）**：`usage-log.md` 是「越用越好用」的唯一事实来源——造/改技能时即生成骨架；**发布后首次使用必须追加一行真实记录，否则下次 PR 评审直接判 P0**。`usage-log.md` 空转 = 「自我蒸馏」空转 = 「越使用越习惯」只是承诺而非事实。三件套就位 ≠ 机制运转。
